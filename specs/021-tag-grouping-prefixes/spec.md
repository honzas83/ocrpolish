# Feature Specification: Tag Grouping Prefixes

**Feature Branch**: `021-tag-grouping-prefixes`  
**Created**: 2026-05-20  
**Status**: Draft  
**Input**: User description: "In this feature, we will organize the tree of tags into three groups: #Topic/... -- tags from category/topic hierarchy #Entity/... -- detected named entities #Tags/... -- assigned tags by LLM The prefixes will be added programmatically after their (unchanged) detection by LLM"

## Clarifications

### Session 2026-05-20
- Q: Should prefixes be hardcoded or configurable? → A: Use global configurable constants in a single file for prefixes (Topics, Entities, Tags).
- Q: Where should the global prefix constants be defined? → A: ocrpolish/data_model.py (Central data definitions).
- Q: What if a prefix constant is set to None? → A: If the constant is None, do not prefix the corresponding group.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Prefixed Taxonomy Tags (Priority: P1)

As a researcher, I want taxonomy-based tags (topics) to be grouped under a `#Topic/` root so that I can easily browse the conceptual hierarchy in Obsidian's tag pane.

**Why this priority**: High. It's the primary way users browse the thematic content of the archive.

**Independent Test**: Process a document with known taxonomy matches and verify that they appear with the `#Topic/` prefix in the output.

**Acceptance Scenarios**:

1. **Given** a document with the topic "Strategy", **When** processed, **Then** the tag should appear as `#Topic/Strategy`.
2. **Given** a hierarchical topic "Nuclear-Deterrence/Doctrine", **When** processed, **Then** it should appear as `#Topic/Nuclear-Deterrence/Doctrine`.

---

### User Story 2 - Prefixed Entity Tags (Priority: P1)

As a researcher, I want detected entities to be grouped under an `#Entity/` root so that all actors and locations are separated from thematic concepts.

**Why this priority**: High. This separates "what" the document is about from "who" or "where" it involves.

**Independent Test**: Process a document with detected entities and verify they have the `#Entity/` prefix.

**Acceptance Scenarios**:

1. **Given** an entity "State/United-Kingdom", **When** processed, **Then** it should appear as `#Entity/State/United-Kingdom`.
2. **Given** a person "Person/Joseph-Luns", **When** processed, **Then** it should appear as `#Entity/Person/Joseph-Luns`.

---

### User Story 3 - Prefixed Assigned Tags (Priority: P2)

As a researcher, I want flat tags assigned by the LLM (not in taxonomy) to be grouped under a `#Tags/` root to distinguish them from the official taxonomy and entities.

**Why this priority**: Medium. Provides a catch-all for keywords while maintaining the clean top-level structure.

**Independent Test**: Process a document with conceptual tags and verify the `#Tags/` prefix.

**Acceptance Scenarios**:

1. **Given** a conceptual tag "Reforger", **When** processed, **Then** it should appear as `#Tags/Reforger`.

---

### Edge Cases

- **Double Prefixing**: System MUST NOT add a prefix if it's already present (e.g., if LLM already output `#Entity/...`).
- **Empty Tags**: System MUST NOT create empty prefixed tags if the underlying tag is empty.
- **Normalization**: Prefixes MUST be applied AFTER normalization to ensure consistent casing and hyphenation.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST prefix all `topic_tags` (taxonomy hierarchy) with `Topic/`.
- **FR-002**: System MUST prefix all `entity_tags` (State, Org, City, Person) with `Entity/`.
- **FR-003**: System MUST prefix all `conceptual_tags` (LLM-assigned flat tags) with `Tags/`.
- **FR-004**: System MUST ensure that the leading `#` is applied only once at the very beginning of the tag string.
- **FR-005**: The prefixing MUST happen programmatically in the processing pipeline, not by changing the LLM prompts.
- **FR-006**: The system MUST be idempotent regarding prefixes; it must not double-prefix tags that already contain the target prefix.
- **FR-007**: If a global prefix constant is set to `None`, the system MUST NOT apply any prefix to the corresponding group of tags.

### Key Entities *(include if feature involves data)*

- **Taxonomy Tag**: A tag derived from the `NATO_themes.yaml` hierarchy.
- **Entity Tag**: A hierarchical tag representing a real-world entity (State, Organization, City, Person).
- **Assigned Tag**: A flat keyword generated by the LLM to capture concepts not covered by the taxonomy.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of tags in the Obsidian vault belong to their configured root hierarchies (e.g., `#Topic/`, `#Entity/`, or `#Tags/`), or remain unprefixed if the corresponding constant is set to `None`.
- **SC-002**: No tags in the Obsidian vault are double-prefixed (e.g., `#Entity/#Entity/...`).
- **SC-003**: Existing tag normalization (hyphenation, casing) is preserved within the new prefixed structure.
- **SC-004**: Tag grouping in the Obsidian Tag Pane correctly reflects the three-tier organization.
